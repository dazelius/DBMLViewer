# AEGIS 기획서 작성 프로세스 규칙

## 트리거
- 사용자가 "Aegis"를 서두에 붙여서 기획서/문서 요청 시 이 규칙을 적용한다.
- "Aegis" 없이 요청하면 기존 방식대로 조사 내용을 표시한다.

---

## 0단계: 목적 확인 질문
사용자가 기획서와 유사한 문서를 요청하면, 먼저 다음을 질문한다:
- "기획서 작성을 위한 문서인지, 정확한 기획 내용을 확인하고 싶은 것인지?"
- 기획서 작성 목적이 아니라면 → 기존 방식대로 조사 내용 표시 (이 규칙 미적용)
- 기획서 작성 목적이라면 → 아래 프로세스 진행

---

## 1단계: 초안 작성
- 기존 방식대로 데이터/코드/컨플루언스 등을 조사하여 문서를 작성한다.
- 작성된 문서를 **"초안"**이라고 명시한다.
- 초안 제공 후, 다음 검증 단계를 수행할지 사용자에게 질문한다.

---

## 2단계: 용어 검증
- 문서에 사용된 '개발용어'에 가까운 용어를 추린다.
- 각 용어에 대해 사용한 의미를 설명하고, 체크리스트 형태로 제공한다.
- 각 항목에 Yes/No로 응답할 수 있도록 한다.
- 용어 변경 니즈가 있으면 사용자가 직접 기입할 수 있도록 안내한다.
- 각 용어마다 유사 의미의 추천 단어도 함께 표시한다.
- 이 단계 완료 후: 초안 수정할지, 다음 검증(3단계)으로 넘어갈지 사용자에게 질문한다.

---

## 3단계: 기획 의도 추가 검증
- 테이블/코드 데이터만으로 작성되어 컨플루언스와 교차검증되지 않은 내용을 추린다.
- 해당 내용을 자연어로 해석하여 체크리스트로 제공한다 (코드 로직은 자연어 설명).
- 사용자에게 기획 의도에 맞는지 Yes/No로 설문받는다.
- 의도와 다르면 정확한 내용을 사용자에게 기입 요청한다.
- 이 단계 완료 후: 초안 수정할지, 최종본으로 확정할지 사용자에게 질문한다.

---

## 검증 중 프로세스 유지 규칙
- 2단계, 3단계 진행 중 바로 문서를 수정하지 않는다. 각 단계별로 "초안 수정 vs 다음 검증" 선택지를 제공한다.
- 사용자가 검증 도중 별도 질문을 하면, 답변 후 반드시 "검증 프로세스를 계속할까요?"라고 질문하여 프로세스가 멈추지 않도록 한다.

---

## 4단계: 초안 수정 (반복)
- 검증 결과를 반영하여 초안을 수정할 때:
  1. 어떤 부분을 수정할 것인지 목록을 먼저 제시한다.
  2. 수정 내용이 검증 결과에 맞는지 체크리스트로 사용자에게 제공한다.
  3. 사용자가 의도와 다르다면 어떻게 다른지 설문받는다.
  4. 사용자가 **최종적으로 전체 문서 수정을 요청**하기 전까지 이 단계를 반복한다.

---

## 연출 관련 내용 특별 규칙
- 머티리얼 색상 변경, FX, 애니메이션 연출 방식 등 **연출 관련 내용**을 작성할 때:
  - 컨플루언스로 교차검증되지 않은 내용은 문서 내에서 눈에 띄는 별도 박스로 표시한다.
  - 박스 제목: **"연출 확인 필요"**
  - 각 항목에 정보 출처를 명시한다: (코드), (데이터), (DataMaster 창작) 등
- 연출 관련 내용에는 2~4단계의 검증/질답/직접 수정 프로세스를 적용하지 않는다.
- 출처 표시 및 연출 관련임을 확인할 수 있도록 표시만 한다.

---

## 연출 확인 필요 박스 형식 (아티팩트 내)
```html
<div style="border:2px solid #f59e0b;background:#1a1600;border-radius:8px;padding:16px;margin:16px 0;">
  <h3 style="color:#f59e0b;margin:0 0 12px;">연출 확인 필요</h3>
  <ul style="color:#fbbf24;">
    <li>내용 설명 <span style="color:#94a3b8;font-size:12px;">(출처: 코드)</span></li>
    <li>내용 설명 <span style="color:#94a3b8;font-size:12px;">(출처: 데이터)</span></li>
  </ul>
</div>
```

---

## 요약 흐름
```
[Aegis 서두] → 0.목적 확인 → 1.초안 작성 → 2.용어 검증 → 3.기획 의도 검증 → 4.수정 반복 → 최종 확정
                                                    ↕                ↕               ↕
                                              (중간 질문 시 답변 후 프로세스 복귀)
                                              (연출 내용은 검증 대신 출처 표시만)
```
